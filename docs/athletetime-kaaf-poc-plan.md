# AthleteTime — KAAF 공개데이터 색인 PoC 설계서

> 작성: Claude (신뢰/인문 도메인) · 갱신: 2026-06-07
> 상태: **설계 단계 (코드 미착수)** — 마스터 로드맵의 S3.
> 🧭 **상위 기준 문서:** `athletetime-data-strategy-master.md` (전략 종합·확정)
> 선행 필수: 마스터 §7 Go 조건 + 선생님 승인
> **B안 검증 추가:** PoC는 person_no를 매칭에만 쓰고 **저장 안 함(폐기)**, 난수 canonicalId만 남기는 흐름까지 검증한다.
> 목적: 본격 구현 전, **소규모·저속·합법** PoC로 (1) person_no 매칭 가능성,
> (2) 우리 데이터와의 매칭률, (3) 데이터 품질, (4) **person_no 폐기 흐름**을 **측정**한다. 측정이 목적이지 수집이 목적이 아니다.

## 1. PoC 범위 (의도적으로 좁게)

| 항목 | 범위 | 이유 |
|---|---|---|
| 대상 사이트 | **`www.kaaf.or.kr` 공개 페이지만** | robots 허용 구역 (R1) |
| 제외 | `result.kaaf.or.kr` (robots 전면차단), pis2 | R1·R10 |
| 표본 크기 | **선수 30~50명** | 통계적 매칭률 추정 최소치, 부하 최소화 |
| 대상 선수 | **성인·현역 공개선수 위주** | R4 미성년 회피 |
| 종목 | 우리 데이터에 풍부한 1~2종목(예: 100m, 멀리뛰기) | 매칭 신호 풍부 |
| 수집 항목 | person_no, 이름, 생년힌트, 연도별 소속, 출처URL | §아키텍처 3.1 |
| 결과물 | **측정 리포트 1건** (코드는 PoC 후 폐기/격리) | "측정이 목적" |

## 2. 수집 정도(正道) — 코드 강제사항

```
- robots.txt 런타임 파싱 → Disallow 경로 자동 제외 (result.kaaf 하드 차단)
- 요청 간격: 최소 2~3초 (jitter 포함), 동시성 1
- 시간대: 야간 배치 권장
- 일일 상한: PoC는 총 ≤ 200 요청
- User-Agent: 서비스명 + 연락처 명시 (예: "AthleteTime-Indexer/PoC (contact: ...)")
- 실패/429/403 시: 즉시 backoff, 재시도 최소화, 로그 남김
- 인코딩: EUC-KR/UTF-8 자동 판별 후 디코딩
```

> 이 제약들은 "성능"이 아니라 **"합법성·예의의 증거"** 다. 느린 게 정답이다.

## 3. 측정 지표 (PoC 성공/실패 판정 기준)

| 지표 | 정의 | 목표 |
|---|---|---|
| **취득률** | person_no를 얻은 표본 / 시도 표본 | ≥ 80% |
| **매칭률** | 우리 athleteKey와 confidence≥0.85 매칭 / 취득 표본 | ≥ 60% (측정치) |
| **오매칭률** | 수동 검증상 잘못 묶인 비율 | ≤ 2% (R6 안전선) |
| **소속이력 완성도** | 연도별 소속이 2개 이상 잡힌 비율 | 측정만 (기대치 미설정) |
| **person_no 안정성** | 동일선수 2회 조회시 동일 person_no | 100% 기대 |
| **요청당 평균지연/차단** | 429/403 발생 여부 | 0건 |

→ 취득률·오매칭률이 기준 미달이면 **person_no 연동 자체를 재검토** (정직한 실패 기준 명시).

## 5. 실행 전 최종 체크리스트 (Go/No-Go)

**모두 ✅ 여야 PoC 코드 착수:**
- [ ] KAAF 사전 협조요청 발송 (R2 영리조항) — 최소한 발송 완료
- [ ] person_no 화면 비노출 원칙 선생님 합의 (아키텍처 §5)
- [ ] 미성년 회피(성인·공개선수 한정) 표본설계 확정
- [ ] robots 준수·저속·일일상한 코드 가드 구현·검증
- [ ] PoC 코드는 격리 디렉터리(`tools/poc/`)에, 서비스 런타임과 분리
- [ ] 수집 데이터는 PoC 종료 후 보관/폐기 정책 결정

**No-Go 신호 (하나라도면 중단):**
- KAAF가 명시적 거부 회신
- robots/약관 해석상 위법 소지 확인
- 미성년 데이터 회피 불가
- 오매칭률이 안전선 초과

## 5.5. 오프라인 PoC 1차 결과 (2026-06-07, 외부수집 전)

실제 KAAF 취득에 앞서, **로컬 인덱스만으로** 동일인 매칭 알고리즘의 거동을 먼저 측정했다.
도구: `tools/identity-poc.js` (네트워크 0, 파일쓰기 0, 표준출력 리포트만). 무해.

### 측정값
| 지표 | 값 |
|---|---|
| 전체 선수 카드 | 16,853 |
| 동명 클러스터(2명 이상) | **3,383개** (전체의 약 20%가 이름 충돌) |
| 평가 후보쌍(100 클러스터) | 6,879 |
| 자동병합 후보(conf≥0.85) | **0 (0.0%)** |
| 사람검토 후보(0.5~0.85) | 1,342 |
| person_no 유출 | ✅ 없음(assert 통과) |

### 핵심 결론 (정량적 근거)
1. **이름+소속만으로는 자동 동일인 판정이 불가능하다.**
   - 1차(흔함 패널티 없음): 김민서 30명·김민지 29명이 0.95로 **과병합** → 명백한 오매칭.
   - 보정(클러스터 크기 흔함 패널티 도입) 후: 흔한 이름은 전부 reject, **자동병합 0%**.
   - → 강한 외부 식별자(person_no) 없이 자동병합하면 **오매칭이 신뢰를 파괴**한다.
2. **이것이 B안(person_no 판단 보조 후 폐기)의 정당화다.**
   - person_no 같은 1:1 식별자가 있어야만 안전한 동일인 판정이 가능.
   - 동시에 그 식별자를 **저장하지 않고 폐기**하면 타인 개인정보 비보유 + 정확성을 모두 취한다.
3. **표기 변형 탐지 여지**: "경기-과천문원중학교" ↔ "경기과천문원중학교"처럼
   같은 소속의 표기 흔들림은 별도 정규화로 잡을 수 있음(향후 개선 포인트).

### 알고리즘 설계 메모(보수성 원칙)
- 기본점 낮게(0.4), 공통 종목 없음 감점, **이름 흔함(클러스터 크기) 강한 감점**.
- "애매하면 reject" — 오매칭 비용 ≫ 미병합 비용. 자동병합 임계 0.85 유지.
- 실제 PoC(외부수집)에서는 person_no 일치를 **결정적 신호**로 추가 → 이때만 자동병합 허용.
