# AthleteTime 경기결과 커버리지 기준

> 상태: 2026-06-25 기준 운영 판단 문서
> 증거 산출물: `.ultra/docs/research/athletetime-coverage-matrix-2026-06-25.md`
> 생성 명령: `node tools/build-coverage-matrix.js --from-year 2010 --to-year 2026`

## 결론

아직 `2010년부터 오늘까지 모든 경기결과`가 다 있는 상태가 아니다.

현재 로컬 구조화 데이터는 `2018-2026` 중심이며, 그중 `2026`은 대회 목록 `60개`에 결과 묶음 `6개`만 있어 명백히 부분 커버리지다. `2010-2017`은 로컬 대회 목록과 결과 묶음이 모두 비어 있다.

## 현재 수치

| 구분 | 값 |
|---|---:|
| 요청 범위 | 2010-2026 |
| 전체 연도 | 17 |
| 로컬 대회목록 보유 연도 | 9 |
| 로컬 결과묶음 보유 연도 | 9 |
| 로컬 목록과 결과 수가 맞는 연도 | 8 |
| 로컬 결과가 부족한 연도 | 1 |
| 시작 전 연도 | 8 |
| 가장 이른 로컬 대회목록 | 2018 |
| 가장 이른 로컬 결과묶음 | 2018 |

## 해석 원칙

`locally_aligned_not_global_proof`는 "우리 로컬 대회 목록 수와 결과 묶음 수가 같다"는 뜻일 뿐이다. 이것을 "그해 전체 대회를 모두 확보했다" 또는 "공식 완전 데이터"라고 표현하면 안 된다.

`partial_local_gap`은 우리 로컬 대회 목록 안에서도 결과가 부족하다는 뜻이다. 2026년은 이 상태다.

`not_started`는 해당 연도에 로컬 대회 목록과 결과 묶음이 모두 없다는 뜻이다. 2010-2017년은 이 상태다.

## 다음 수집 순서

1. `2010-2017` 대회 목록부터 만든다.
2. 각 대회별 공식 결과 파일을 찾는다: PDF, XLS, XLSX, HWP.
3. data.go.kr 공개 파일을 import해 대회명, 기간, 장소, 종목, 참가자, 결과 후보를 보강한다.
4. KAAF 기준 기록 페이지는 결과 원본이 아니라 검수용 reference table로 사용한다.
5. `result.kaaf.or.kr`처럼 차단 정책이 있는 경로는 우회하지 않는다. 공식 제공 파일, 공개 데이터, 수동 업로드, 협조 요청으로 대체한다.

## 출처 유형 4단계

| 유형 | 의미 | 사용 방식 |
|---|---|---|
| `public_data_import` | 공공데이터포털 파일 데이터 | 라이선스와 항목 정의를 남기고 batch import |
| `download_file` | 공식 페이지의 PDF/XLS/HWP 첨부파일 | 원본은 비공개 보관, 파싱 결과만 서비스 반영 |
| `fetch_reference_table` | KAAF TOP기록, 부별기록, 연도별 기록 현황 같은 조회표 | 결과 원본 대체가 아니라 검수·비교 기준 |
| `blocked` | robots/접근정책상 자동 수집 금지 경로 | 우회 금지, 협조 요청 또는 공식 파일 확보 필요 |

## 운영 문구

권장:

> AthleteTime은 공개 경기결과와 공식 제공 파일을 모아 정리하고 있습니다. 현재 데이터는 2018-2026 중심이며, 2010-2017과 2026 일부 결과는 계속 보강 중입니다.

금지:

> 2010년부터 모든 육상 경기결과를 보유하고 있습니다.

금지:

> 공식 전체 기록 DB입니다.

## 갱신 방법

```bash
node tools/build-coverage-matrix.js \
  --from-year 2010 \
  --to-year 2026 \
  --generated-at 2026-06-25T00:00:00.000Z \
  --out-json .ultra/docs/research/athletetime-coverage-matrix-2026-06-25.json \
  --out-md .ultra/docs/research/athletetime-coverage-matrix-2026-06-25.md
```

새 자료를 import한 뒤에는 이 명령을 다시 실행해 커버리지 수치가 실제로 개선됐는지 확인한다.
