# AthleteTime — 본인 요청 기반 "내 기록 묶기" 흐름 기획

> 작성: Claude (신뢰/인문 도메인) · 2026-06-08
> 상위 기준: `athletetime-identity-roi-decision.md`(§3.3) · `athletetime-data-strategy-master.md`
> 상태: **기획 단계 (코드 미착수)** — Go 조건 + 선생님 승인 후 구현
> 목적: 자동 병합의 오결합 위험 없이, "내 기록을 한 번에 보고 싶다"는 니즈를
>       **본인 의사 확인을 거쳐** 안전하게 푼다.

---

## 1. 왜 자동이 아니라 "본인 요청"인가

| 자동 병합 | 본인 요청 병합 |
|---|---|
| 흔한 이름 동명이인(4,735명) 오결합 위험 | 본인이 직접 고른 기록만 묶음 → 오귀속 구조적 차단 |
| 미성년 타인 기록 잘못 붙임 = R6/R4 | 본인(또는 보호자) 의사 = 정당성 확보 |
| "우리가 단정" = 색인자 정체성 위반 | "사용자가 선택" = 색인자 + 사용자 주권 |

→ 자동 병합이 풀려는 문제(한 사람 = 한 묶음)를, **단정의 주체를 우리에서 본인으로 옮겨** 해결.

---

## 2. 핵심 원칙

1. **기본 상태는 "구분 표시"**: 묶기 전엔 동명이인 카드가 나란히 보인다(현행 유지).
2. **묶기는 본인의 명시적 선택**: "이 기록들이 내 것"이라고 사용자가 고른다.
3. **본인 확인을 거친다**: 단순 클릭이 아니라 약한 수준이라도 본인성 신호를 받는다(§4).
4. **묶어도 출처·동명이인 고지는 유지**: 묶음 카드에도 "본인 신고 기반" 라벨.
5. **타인 기록 귀속 차단**: 한 묶음에 들어간 기록은 다른 사람이 가져갈 수 없다(소유 1회성).
6. **철회 가능**: 언제든 묶음 해제/기록 제거 요청 가능(R5 삭제 SLA와 연결).
7. **person_no 미사용**: 본인 묶기는 외부 식별자가 필요 없다(본인이 근거).

---

## 3. 사용자 흐름 (텍스트 와이어프레임)

```
[검색] "홍길동"
   ↓
[결과] 같은 이름 후보 카드 N개 (각 카드 = athleteKey 묶음)
   │   "이름이 같은 다른 선수일 수 있어요 · 소속/연도 확인"
   ↓  (사용자가 "내 기록 묶기" 진입)
[1. 후보 고르기]
   - "내 기록인 카드"를 체크 (여러 개 선택 가능)
   - 시스템이 'shadow cluster' 추정 후보를 미리 ✔ 제안(opt-in) — "같은 선수로 추정돼요"
   - 사용자가 추가/제외 자유롭게 조정
   ↓
[2. 본인 확인]  (§4 단계별 옵션 중 택1)
   ↓
[3. 묶음 생성]
   - 내 기록 타임라인(연도·소속 변화)으로 표시
   - 라벨: "본인 신고 기반 묶음 · 공식 기록 아님 · 출처 표시"
   ↓
[4. 관리]
   - 기록 추가/제외, 묶음 해제, 삭제 요청 언제든 가능
```

## 4. 본인 확인(authenticity) 단계 — 강도별 옵션

> 초기엔 **약한 신뢰(L1)** 로 시작하고, 민감/분쟁 시 상향. 과도한 PII 수집은 피한다(R3 최소화).

| 레벨 | 방법 | 신뢰도 | 수집 정보 | 비고 |
|---|---|---|---|---|
| L1 | 로그인 계정 + "내 기록임" 자가 선언 | 낮음 | 계정만 | 초기 기본. 묶음에 "자가 신고" 라벨 |
| L2 | 추가 식별 질문(예: 특정 대회 참가 여부 등 본인만 아는 사실) | 중간 | 답변(저장 최소) | 흔한 이름 분쟁시 |
| L3 | 보호자/본인 연락 + 수동 검토(매거진·고가치) | 높음 | 최소 연락정보(검토 후 폐기 지향) | 예외적 |

- **미성년**: L1만으로 자동 노출하지 않는다. 보호자 경로(L3) 우선, 기본은 비공개 묶음.
- 수집한 본인확인 정보는 **묶음 권한 판단에만** 쓰고, 불필요해지면 폐기(B안 정신 계승).

## 5. 데이터 모델 (제안 — 구현은 추후)

```jsonc
// 본인 신고 묶음 (자동 canonicalId와 분리: 출처가 "사용자")
{
  "claimId": "claim_xxx",          // 우리 발급 난수
  "ownerAccountId": "acc_...",     // 묶은 사용자 계정(본인성 근거)
  "claimedAthleteKeys": ["<key1>", "<key2>"],  // 본인이 고른 athleteKey 묶음
  "authenticityLevel": "L1",       // §4
  "status": "active",              // active | revoked
  "createdAt": "...",
  "label": "self_claimed",         // 화면에 "본인 신고 기반" 표기
  // ❌ person_no 없음. ❌ 생년월일 없음.
}
```
- **자동 추정(shadow cluster)** 과 **본인 신고(claim)** 는 별도 레이어. 충돌 시 본인 신고 우선.
- 한 athleteKey가 둘 이상의 active claim에 들어갈 수 없다(소유 1회성, §2-5).

## 6. 화면 카피 원칙 (신뢰 톤)

| 상황 | 카피(안) | 회피 |
|---|---|---|
| 후보 나열 | "이름이 같은 다른 선수일 수 있어요. 소속·연도를 확인해 주세요." | "동일인 확정" |
| 추정 제안 | "같은 선수로 추정되는 기록이에요. 직접 확인해 주세요." | "당신의 기록입니다"(단정) |
| 묶음 라벨 | "본인 신고 기반 묶음 · 공식 기록 아님" | "공식/검증된 프로필" |
| 미성년 | (기본 비공개) "보호자 확인이 필요한 묶음이에요." | 자동 공개 |

- 금칙: 공식/랭킹/검증/예측/평가 + 미래약속(곧/예정). "추정"은 항상 "직접 확인" 안내와 함께.

## 7. 단계적 롤아웃

1. **(지금) 구분 표시 + 동명이인 경고 강화** — 묶기 기능 없이 안전. Claude UX 작업.
2. **shadow cluster 추정 표기** — "같은 선수로 추정" *제안만*, 묶지 않음. Codex 엔진 + Claude 카피.
3. **L1 본인 묶기(opt-in)** — 로그인 기반 자가 선언. 계정 시스템 연동 필요.
4. **L2/L3 + 미성년 보호자 경로** — 분쟁/민감 케이스. Go 조건·법적검토 후.

각 단계 독립 롤백 가능. 1단계는 외부 데이터·계정 의존 0.

## 8. 미해결·확인 필요
- 계정 시스템(현 `/api/auth`)과 claim 연동 범위 — 3단계 진입 시 설계.
- L2 식별 질문의 PII 최소화 기준 — R3와 함께 확정.
- 미성년 보호자 인증 수단 — 법적검토서와 연계.
- shadow cluster 추정 임계값 — `identity-architecture.md`의 0.85/0.6과 정합.
