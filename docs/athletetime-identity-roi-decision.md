## 0. 한 줄 답 (✅ 확정)

**"전체를 고유번호로 검증해 다듬는다"는 하지 않는다.** 비용 대비 효과가 나쁘고,
우리 정체성(공식 기관 아님, 색인자)과도 충돌한다. 대신 **검색 시점에 동명이인을
'합치지 않고 나란히 보여주는'** 방식으로 간다. 고유번호는 *전체 정제용*이 아니라
*소수의 고가치 선수*(검색 빈도 상위·본인 요청·매거진 콘텐츠용)에만 제한적으로 쓴다.

> ⚠️ **선생님 조정(더 보수적):** 진학 712명조차 **확정 자동병합 금지.**
> `추정 묶음` 또는 `opt-in 안전병합`으로만 둔다. "내 기록 한 번에 모으기" 니즈는
> **자동 병합이 아니라 본인 인증/요청 기반**으로 푼다.

## 0-A. ✅ 승인된 최종 결정 (2026-06-08, 선생님)

1. **동명이인은 합치지 않는다.**
2. **진학/소속변경 712명도 즉시 확정 병합 금지** → `추정 묶음` / `opt-in 안전병합`만.
3. **고유번호 Layer3는 전체 ❌** → 검색 빈도 높은 선수·본인 요청·매거진 콘텐츠용 고가치 선수에만 제한.
4. **`person_no`는 저장하지 않는다.** 판단 후 즉시 폐기(B안).
5. **파싱쓰레기 이름(예 `"0"`)은 검색/통계/인사이트 색인에서 제외.** 원본 결과표에는 보존.

> 근거: 우리 문제의 본질은 "진학"이 아니라 "동명이인"(4,735 vs 712, 6.6배).
> 이름 기반 자동 병합은 **거의 하면 안 된다.** 미성년 비율이 높아 오결합은
> 단순 오류가 아니라 "타인의 기록을 다른 사람에게 붙이는 문제"가 된다.

**불편하지만 감수하는 결과(의도된 트레이드오프):**
- 한 선수의 기록이 여러 카드로 나뉘어 보일 수 있다. → **본인 인증/요청 기반 "내 기록 묶기"** 로 해결.
- 초기에 "내 기록 한 번에 모아줘"가 완벽히 풀리지 않는다. → 안전을 위한 의도된 선택.

## 1. 실측: 문제의 진짜 크기 (2026-06-08, 91,004행 스캔)

| 지표 | 값 | 의미 |
|---|---|---|
| 총 행(이름 보유) | 91,004 | |
| 유효 고유 이름(파싱쓰레기 제외) | 10,946 | |
| 이름 같고 소속 2개 이상 | **5,538명 (46.5%)** | 진학+동명이인+대항전 혼재 |
| **같은 해 동시 다소속(동명이인 강한 신호)** | **4,735명 (43.3%)** | 진학으로 설명 불가 |
| 진학/이적으로 자연 설명(연도당 1소속) | **712명** | "묶어도 안전"한 케이스 |
| **동명이인 의심 이름이 차지하는 행** | **77,286 / 89,646 (86.2%)** | 🔴 |

### 1.1 결정적 통찰
- 우리가 두려워한 "진학 추적"은 **712명**뿐. 전체의 일부.
- 진짜 큰 문제는 **동명이인**: 같은 해에 동시에 여러 소속을 가진 이름이 **4,735명**으로
  진학 케이스의 **6.6배**.
- "김민지" 56종 소속, "김도연" 52종 — 이건 진학이 아니라 **수십 명의 다른 김민지**.
- → **이름만으로는 86.2%의 행을 신뢰성 있게 한 사람으로 묶을 수 없다.**

### 1.2 그래서 고유번호가 "전체 정제"를 풀어주나?
**아니다 — 부분만 풀고, 새 리스크를 만든다.**
- 고유번호가 있으면 4,735명의 동명이인을 *이론상* 분리할 수 있다. 그러나:
  - 우리 데이터엔 고유번호가 **없다.** 외부(KAAF 등)에서 4,735명 × 평균 다수 후보를
    매칭해야 하는데, 이름만 같은 후보가 수십 명이라 **매칭 자체가 동명이인 문제와 동형**.
  - 매칭 신뢰도 0.85 미만이면 보류해야 함(명예훼손 방어, R6). 흔한 이름일수록 신뢰도가
    낮아져 **정작 가장 문제인 케이스가 자동화 안 됨.**
  - 잘못 합치면(다른 김민지를 한 명으로) → **타인의 기록을 그 사람 것으로 단정** = R6 위험.

## 2. 비용 vs 효과

### 2.1 "전체 고유번호 검증·정제" 비용
| 항목 | 비용 |
|---|---|
| 외부 식별자 확보 | 4,735+ 동명이인 후보 전수 대조. 흔한 이름은 후보 폭증 |
| 매칭 엔진 정밀도 | 흔한 이름에서 confidence 낮음 → 수동 검수 대량 발생 |
| 법적/윤리 | person_no 대량 참조(B안이라 폐기해도) + 오결합 명예훼손 노출 |
| 미성년 | 동명이인의 58%가 미성년(초·중·고) → 정밀 식별은 R4와 정면 충돌 |
| 유지보수 | 매년 신규 대회로 동명이인 계속 증가 → **끝없는 정제 부채** |

### 2.2 효과
- 712명(진학)만 깔끔하게 풀림 — 이건 **고유번호 없이도** 풀 수 있음(§3).
- 4,735명(동명이인)은 고유번호로도 *부분만*, 그것도 위험을 안고 풀림.
- **결론: 가장 비싼 부분(동명이인)이 가장 효과가 낮다.** 전형적인 나쁜 ROI.

## 3. 권고: "정제(merge)" 대신 "구분 표시(separate & disclose)"

우리는 **공식 기관이 아니라 색인자**다. 색인자의 올바른 자세는
"하나로 단정"이 아니라 **"같은 이름의 서로 다른 기록을 정직하게 나란히 보여주는 것"**이다.

### 3.1 즉시(저비용·고효과) — 고유번호 불필요
1. **동명이인 비합치 원칙 확정**: athleteKey(`name|team|연도근접`)로 묶되,
   **합쳐서 한 명으로 단정하지 않는다.** 검색 결과에 후보를 카드로 나열(이미 구현됨).
2. **동명이인 경고 강화**: "이름이 같은 다른 선수일 수 있어요 · 소속/연도 확인" (이미 노출, 추가 강화 예정).
3. **파싱 쓰레기 격리**: 이름이 `0`·숫자·깨진 문자열인 행(예: "0" 306행)을
   **검색/통계/인사이트 색인에서 제외**, **원본 결과표에는 보존** → Codex 도메인.
4. **712명 진학 케이스 = `추정 묶음`/`opt-in 안전병합`만** (⚠️ 확정 자동병합 ❌):
   "연도당 소속 1개"여도 자동으로 한 명으로 확정하지 않는다.
   - 기본: **shadow cluster**(내부 후보 묶음)로만 계산, 화면엔 "같은 선수로 추정되는 기록"으로 *제안* 표기.
   - 사용자가 동의(opt-in)할 때만 묶어서 보여준다. 기본 노출은 여전히 "구분 표시".

### 3.2 나중에(Go 조건 충족 후) — 고유번호 *제한적* 사용
- **전체 정제 ❌ / 고가치 소수 ✅**: 범위를 명확히 제한한다 —
  ① 검색 빈도 높은 선수 ② 본인 요청 ③ 매거진 콘텐츠용 고가치 선수.
- person_no는 B안대로 **판단 후 폐기**(저장 안 함). canonicalId(난수)만 보유.
- 임계값 미달(흔한 이름)은 자동 병합하지 않고 "구분 표시"로 graceful fallback.

### 3.3 "내 기록 한 번에 모으기"는 자동이 아니라 **본인 요청 기반**
- 자동 병합으로 풀지 않는다(오결합 위험). 대신 본인 인증/요청 흐름으로:
  - 본인이 "이 기록들이 내 것"이라고 선택 → 본인 확인 → 묶기.
  - 출처·동명이인 경고를 유지하며, 타인 기록 오귀속을 구조적으로 차단.
- 기획 문서: `athletetime-self-claim-record-grouping.md`(Claude 작성 예정).

### 3.4 🚫 절대 하지 않을 것 (가드)
- 흔한 이름 4,735명을 자동 병합해 "한 명"으로 단정 — R6/R4 위험, 색인자 정체성 위반.
- **진학 712명도 확정 자동병합 금지** — 추정/opt-in만.
- **전체 person_no 정제 착수 금지** — 이 결정 문서가 단일 기준.
- person_no를 DB에 저장 — B안 위반. 판단 후 즉시 폐기.
