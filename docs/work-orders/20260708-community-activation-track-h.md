## 0. 배경과 목표 (반드시 읽고 시작)

육상인들에게는 커뮤니티/대화 창구가 없다. 애타가 그 창구가 된다.
레퍼런스는 **에브리타임**(검증된 소속 + 익명 = "우리끼리" 안전감)과
**침하하**(개추→베스트 승격, 가벼운 톤, 매일 들어올 이유)다.

육상판 특수성 3가지를 모든 구현 판단의 기준으로 삼는다:

1. **좁은 바닥**: 기록·소속 한 줄로도 특정된다 → 익명은 필수, 고정 닉네임도 위험.
2. **저격 리스크**: "OO대 코치가~" 글 하나가 분쟁이 된다 → 신고/블라인드가 런칭 조건.
3. **미성년 다수**: 중·고 선수 핵심 사용자 → 금칙어 필터 + 자동 블라인드 필수.

**익명의 구조**: 겉은 완전 익명, 서버는 `anonymousId`/userId 보존(제재 가능).
에브리타임과 동일한 이중 구조다. 현재 코드의 anonymousId 레일을 그대로 쓴다.

**신뢰 원칙(WORKFLOW.md §3) 그대로 적용**: 공식/인증/검증/랭킹 단어 금지,
가짜 권위 금지(역할 배지는 "자기 선언"임을 명시), 개인정보 노출 금지.

## H-1a. 게시판 익명 번호링 (에브리타임식)

### 데이터 모델

- `posts`에 `is_anonymous BOOLEAN DEFAULT TRUE` 추가 (기본값 익명 — 에타와 동일).
- `comments`에 `anon_no INTEGER` 추가.
- 번호링 규칙 (에타 방식 그대로):
  - 글 작성자가 익명이면 표시명 `글쓴이`.
  - 댓글 작성자에게는 **글 단위**로 익명 번호를 부여: 같은 글 안에서 같은
    anonymousId가 다시 댓글 달면 같은 번호 유지 (`익명1`, `익명2`, …).
  - 글 작성자가 자기 글에 댓글 달면 번호 대신 `글쓴이` 배지.
  - 번호 매핑은 글별 `post_anon_map(post_id, user_id, anon_no)` 테이블(또는
    comments 삽입 시 트랜잭션으로 MAX+1 계산). **글이 다르면 번호도 리셋** —
    글 간 추적 불가가 핵심이다.
- 익명 해제 옵션: 글 작성 시 "닉네임으로 쓰기" 체크박스(기본 해제).
  익명 글의 작성자 실닉네임은 **API 응답에서 절대 내려보내지 않는다**
  (프론트 숨김이 아니라 서버에서 author 필드 자체를 '글쓴이'/'익명N'으로 치환).

### API 계약

- `GET /api/posts/:id` 및 목록: `author`가 익명 글이면 `"글쓴이"`,
  `user_id`/`anonymous_id`/실명 필드 미포함. 계약 테스트로 잠근다.
- `POST /api/posts`, `POST /api/comments`: `isAnonymous` 수용 (기본 true).
- 수정/삭제 권한: 기존 anonymousId+비밀번호 레일 유지.

### UI 스펙 (확정)

- 작성자 표기: `글쓴이`는 `text-brand`(#0D5F5A) + 소형 배지, `익명N`은
  `text-muted-foreground`. 아바타는 이니셜 없는 중립 원형(익명 실루엣) 단일.
- 글쓰기 페이지: 제목 입력란 위에 토글 라인
  `[✓] 익명으로 작성 (기본)` — 해제 시 현재 닉네임 미리보기 표시.
- 익명 안내 문구(작성 페이지 하단 고정, 정확히 이 문구):
  `익명 글도 커뮤니티 규칙 위반 시 제재될 수 있어요. 특정인 저격·비방은 금지됩니다.`

## H-1b. 신고 → 자동 블라인드 + 금칙어 (런칭 조건)

### 신고 시스템

- 새 테이블 `reports(id, target_type 'post'|'comment'|'chat', target_id,
  reporter_anonymous_id, reason_code, detail, created_at)`.
- 사유 코드(고정): `저격·비방` / `개인정보 노출` / `음란·불쾌` / `도배·광고` / `기타`.
- 동일 대상 **고유 신고자 3명** 도달 시 자동 블라인드:
  - 글/댓글: 본문을 `신고 누적으로 블라인드된 게시물입니다.`로 치환 표시
    (원문은 DB 보존 — 정통망법 임시조치 대응·운영자 검토용).
  - 자기 글 중복 신고 방지, 같은 신고자 중복 카운트 방지.
- 운영자 큐: `GET /api/admin/reports` (기존 admin 인증 레일 사용) —
  블라인드 해제/영구 삭제/작성자 이용 제한 액션.
- 이용 제한: `users`에 `muted_until TIMESTAMP` — 제한 중 글/댓글/채팅 발신 403.

### 금칙어 + 저격 방어

- `backend/utils/contentFilter.js` 신설:
  - 금칙어 리스트(욕설·성적 표현·차별 표현)는 별도 JSON으로 분리, 운영자가 갱신 가능.
  - 1차 필터: 금칙어 포함 시 저장 거부 + 안내
    `커뮤니티 규칙에 맞지 않는 표현이 포함되어 있어요.`
  - **저격 패턴 경고(차단 아님)**: 본문에 실명 추정 패턴
    (`OOO 코치`, `OOO 감독`, `OO대 OOO` 등 호칭+이름 조합) 감지 시
    제출 전 확인 모달: `특정인에 대한 글은 명예훼손 분쟁으로 이어질 수 있어요. 게시할까요?`
    — 게시 자체는 허용(과차단 방지), 감지 로그만 남긴다.
- 채팅 메시지에도 동일 필터 적용 (websocket 핸들러에서 발신 전 검사).

### 커뮤니티 규칙 페이지

- `/community/rules` 정적 페이지 신설. 핵심 5개 조항:
  1. 특정인 저격·비방 금지 (선수·지도자·학부모 모두)
  2. 개인정보(실명·소속·연락처) 노출 금지
  3. 익명은 보호 장치이지 면책이 아님 — 위반 시 제재
  4. 신고 3회 누적 시 자동 블라인드, 운영자 검토 후 조치
  5. 홍보·도배 금지
- 첫 글 작성 시 1회 규칙 동의 모달 (localStorage 플래그).

## H-1c. 채팅 개편: 랜덤 닉네임 + DB 저장 + 방 정리

### 방 구조 (확정)

- 상시 방은 **`자유수다` 1개만** 유지. 기존 training/race/injury 방 제거
  (밀도 집중 — 빈 방 3개가 커뮤니티를 죽인다).
- 라이브방은 H-2a에서 대회 데이터로 동적 생성.

### 랜덤 닉네임 (확정)

- 입장 시 닉네임 입력 대신 **자동 부여 + 리롤 버튼**:
  - 형식: `{수식어} {동물/육상용어} {2자리수}` — 예) `질주하는 치타 42`,
    `역주하는 바람 07`, `막판스퍼트 가젤 88`.
  - 수식어 풀 20개+, 명사 풀 20개+ (육상 감성: 스타트, 스퍼트, 바통, 트랙,
    치타, 가젤, 영양, 바람 등). 조합 400+ × 00–99.
  - 방 내 중복 시 숫자 재추첨. 리롤 무제한(입장 전), 입장 후 변경 불가.
- 세션 유지: 기존 sessionStorage 레일 유지하되 키에 방 스코프 추가 —
  **방이 다르면 닉네임도 다르게** (채팅 간 추적 방지, 에타 익명 철학 일관).

### 메시지 저장

- 새 테이블 `chat_messages(id, room_id, nickname, user_key_hash, body,
  created_at)` — `user_key_hash`는 anonymousId의 해시(제재용, 원문 미저장).
- 입장 시 최근 **200개** 히스토리 로드 (현행 메모리 50개 대체).
- 보존 정책: 30일 경과 메시지 배치 삭제 (가벼운 운영).

### UI 스펙 (확정)

- 접속자 표기: **10명 미만이면 인원수 숨김**, 대신 `오늘 참여 N명`(누적 고유
  닉네임 수) 표시. 10명 이상부터 `현재 N명` 노출.
- 입장 화면: 카드 중앙에 랜덤 닉네임 크게 표시 + `🎲 다른 이름` 버튼 +
  `입장하기` 버튼. 닉네임 직접 입력 UI는 제거.
- 메시지: 내 메시지 우측 `bg-brand`/흰 글자, 타인 좌측 `surface-2` 버블,
  닉네임은 버블 위 11px `muted-foreground`. 시스템 메시지는 중앙 정렬 캡슐.
- 신고: 메시지 롱프레스/우클릭 → `신고` (H-1b 레일 연결).

## H-2a. 대회 라이브방 자동 개설 (우리만의 무기 1)

에타가 시험기간에 터지듯, 육상 커뮤는 **대회 날** 터지게 만든다.

### 동작 규칙

- 소스: 기존 대회 일정 데이터(competitions). 대회 시작일 **전날 18:00**에
  라이브방 자동 개설, 종료일 **다음날 24:00**에 자동 보관(읽기 전용) 처리.
- 방 이름: `🔴 {대회명} 라이브` (예: `🔴 전국실업육상경기대회 라이브`).
- 동시 개최 대회가 여러 개면 방도 여러 개. 개설/보관은 서버 크론(또는
  요청 시점 lazy 평가 — 구현 단순한 쪽 선택, 단 테스트 가능해야 함).
- 보관된 방의 메시지는 30일 보존 정책을 따른다.

### UI 스펙 (확정)

- 채팅 사이드바: `라이브` 섹션(개최 중 방, 🔴 + 방명 + 참여 수) 상단,
  `상시` 섹션(자유수다) 하단. 라이브방 없으면 섹션 자체 숨김.
- **메인 페이지 배너**: 라이브방이 1개 이상 열려 있으면 메인 상단에
  가로 배너 노출 — `🔴 LIVE {대회명} 같이 보기 →` 클릭 시 해당 방 입장.
  `bg-brand-ink`(#07302E) 바탕 + 흰 글자 + 🔴 펄스 애니메이션(CSS only).
  라이브방 없으면 배너 미노출(자리도 차지하지 않음).
- 커뮤니티 페이지 상단에도 동일 배너 재사용 (공용 컴포넌트로).

## H-2b. 결과 속보 봇 (우리만의 무기 2)

수확 파이프라인이 잡은 결과를 커뮤니티에 흘려 대화 떡밥을 만든다.
침하하도 에타도 못 하는 것이다.

### 동작 규칙

- 트리거: 대회 결과 데이터가 신규 적재될 때(수확 파이프라인 후처리 훅).
- 발신 대상: 해당 대회 라이브방(열려 있으면) + `대회` 카테고리 게시판 자동 글.
- 메시지 포맷(확정, 신뢰 원칙 준수 — "공식" 단어 금지):
  - 채팅: `📋 방금 결과가 올라왔어요 — {대회명} {종목} · 자세히 보기 →`
  - 게시판 글: 제목 `[결과] {대회명} {종목}`, 본문에 상위 3위 표
    (순위/이름/소속/기록) + 기록 페이지 딥링크. 작성자 표시 `애타봇`
    (전용 시스템 계정, 아이콘 🤖, 사용자와 시각적으로 구분).
- 도배 방지: 같은 대회는 **종목 묶음 단위**로 최대 30분에 1회 발신
  (큐잉 후 묶어서 1개 메시지/글로).
- 봇 글에는 투표/신고는 가능하되 봇이 베스트에 오르지는 않게 제외 플래그.

## 애타벤 (베스트 승격 — 침하하식)

- 승격 조건: 게시글 **순추천(추천−비추천) 10** 도달 시 `is_best=true` +
  `best_at` 기록. 되돌림 없음(한 번 애타벤은 유지 — 침벤과 동일).
- 초기엔 임계 10으로 시작하되 상수로 분리(`BEST_THRESHOLD`) — 유저 규모
  따라 조정 가능하게.
- 기존 CommunityBestStrip(전체/주간/월간/명예의 전당)에 **`애타벤` 탭 추가**:
  is_best 글을 best_at 역순으로. 명예의 전당과의 차이 = 명예의 전당은
  조회·추천 종합 상위, 애타벤은 임계 통과 전체 아카이브.
- 승격 순간 글 상단에 배지: `🔥 애타벤` (`bg-accent-100 text-accent-700`).
- 글 목록에서도 제목 앞 🔥 아이콘.

## 공통 규칙

1. **신뢰 원칙**: 사용자 노출 문구에 공식/인증/검증/랭킹 금지. 이 지시서의
   확정 문구를 그대로 사용하고, 새 문구가 필요하면 PR 코멘트로 질문.
2. **개인정보**: 익명 글·채팅의 서버측 식별자(anonymousId, user_key_hash)는
   어떤 API 응답에도 포함 금지. 계약 테스트로 잠근다.
3. **PR 단위**: H-1a / H-1b / H-1c / H-2a / H-2b 각각 별도 PR.
   각 PR 본문에 스크린샷(또는 Netlify 프리뷰 URL) + 테스트 결과 필수.
4. **DB 마이그레이션**: schema.sql 직접 수정 + 기존 배포 DB용 idempotent
   마이그레이션 스크립트(`backend/database/migrations/`) 동시 제공.
5. **데이터 트랙과 병행**: 트랙 A(레거시 정규화)·F(2026 시즌)는 이 트랙과
   독립적으로 계속 진행한다. 커밋이 섞이지 않게 브랜치 분리.
6. 매 PR 전 main 동기화, 작은 PR, 증적 필수 — 기존 규칙 동일.
