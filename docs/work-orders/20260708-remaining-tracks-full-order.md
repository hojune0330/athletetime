## 트랙 A: 레거시 정규화 — xls/xlsx 372파일 (P1, 최우선, 이미 지시됨)

선행 지시서 과업 3 그대로 진행한다. dedup 레일(PR #31)이 깔렸으므로 즉시 착수 가능.

- 각 묶음 완료 시 DoD(스팟체크 2–3개 대회, 실패 리포트, year-checklist 갱신,

## 트랙 B: 레거시 정규화 — pdf/hwp 56파일 (P2, 트랙 A 종료 후)

- pdf 39개: 텍스트 추출 가능 여부 먼저 분류 (텍스트 레이어 有/스캔본).
  텍스트 레이어 있는 것만 파싱, 스캔본은 `blocked` 처리하고 목록만 남긴다 (OCR은 이번 범위 밖).
- hwp 17개: `hwp5txt` 등 변환 시도. 실패 시 `blocked` + 사유 기록.
- 출력 스키마·DoD는 트랙 A와 동일. 파싱률이 낮아도 무리하지 말고 격리 우선.

## 트랙 C: TOP100 보류 7,745행 — 팀 종목 스키마 (P2)

계주·역전경기·쉼표 다중선수 행을 서비스에 노출하기 위한 최소 스키마:

1. `data/manual/kaaf-top100`의 held 행을 분석해 유형 분류
   (4x100mR/4x400mR 계주 vs 역전경기 vs 기타).
2. 팀 기록 모델: `teamName + members[] + event + record + date + competition`.
   멤버 이름은 쉼표 분리하되, **개인 선수 검색 인덱스에는 넣지 않는다**
   (동명이인 오합산 위험 — 별도 팀 기록 화면/섹션에서만 노출).
3. 프론트: 선수 프로필이 아닌 별도 "팀 기록" 표시 위치를 제안하고
   (예: 기록 조회 내 릴레이 탭), 구현 전 PR 코멘트로 확인 요청.
4. 노출 문구는 신뢰 원칙 준수 ("KAAF 공개 기록" 계열).

## 트랙 D: 해외 769행 외부 확인 파이프라인 (P2, 수동 작업 지원 도구)

`needs_external_confirmation` 잔여 769행 처리:

1. 769행을 대회별로 그룹핑한 운영자 작업 큐 생성
   (`docs/data-candidates/batches/`에 새 배치 또는 기존 배치 확장).
2. 대회별 외부 공식 소스 후보 URL 매핑 (World Athletics results,
   JAAF, 대만육협, HOKUREN 공식 사이트 등) — 자동 크롤링 금지, URL 목록만.
3. 운영자가 확인 완료 표시하면 `source_verified`로 승격하는
   `tools/promote-kaaf-top100-review-status.js` 확장 (`--confirm <sourceRowId>` 방식).
4. 상위 가치 기록(한국신·대회신 recordType 행) 우선 정렬.

## 트랙 E: 2019–2026 백필 체크리스트 마감 (P2, 가벼움)

2018–2026은 이미 정규 데이터가 서비스 중이므로 원장 정리만 남았다:

1. 각 연도 KAAF 일정 페이지를 훑어 결과첨부를 비공개 볼트에 수집
   (기존 `tools/harvest-kaaf-schedule-results.js` + 국내/국제 페이지).
2. source-ledger 행 추가, year-checklist 상태 갱신.
3. 이미 `data/results`에 있는 대회와 대조해 **누락 대회 목록**을 리포트
   (특히 2026 트랙 시즌 — 현재 로드 6개뿐이므로 누락이 많을 것).

## 트랙 F: 2026 트랙 시즌 결과 수집 (P1, 트랙 E와 연동)

- 트랙 E에서 발견된 2026 신규 결과첨부를 정규화해 `data/results/2026.json`에 추가.
- 시즌 진행 중이므로 이 트랙은 **반복 작업**: 월 1회 이상 재수집 권장.
- 스키마·DoD는 트랙 A와 동일.

## 트랙 G: 서비스 카피·커버리지 정합화 (P1, 각 트랙 완료 시 즉시)

- 데이터 커버리지가 바뀔 때마다: RecordsPage 안내문, about/data 페이지,
  coverage-matrix 갱신이 한 커밋에 같이 가야 한다.
- "공식·인증·검증·랭킹" 금지 원칙 유지. 표현은 "모은 공개 기록" 계열.

## 공통 규칙 (전 트랙)

- 원본 파일 커밋 금지 / 식별자·생년 출력 금지 / dataRequestService 마스킹 준수.
- PR은 트랙·연도 단위로 작게. 한 PR에 여러 트랙 섞지 않기.
- 각 PR 본문: 변경 요약 + 테스트 결과 + 증적 경로 + (해당 시) 스팟체크 결과.
- 판단이 필요한 지점(스키마 변경, 새 노출 표면)은 구현 전 PR 코멘트로 질문.
- 매 PR 전 main 최신화 (Claude 트랙과 병렬 진행 중).
