## 0. 이 문서가 답하는 것

선생님과 4차례에 걸쳐 논의·우려한 내용을 하나로 종합한다:
1. 무엇을 저장하고(기록 ✅) 무엇을 안 남기는가(민감정보 ❌)
2. person_no를 "판단 후 폐기"하는 B안 설계
3. 해외 사례·공공데이터를 기반으로 한 정당성 확보
4. 법적 문제를 피하는 다중 방어선
5. 해외 데이터 수집/관할 고려
6. 단계별 실행 로드맵

> ⚠️ **전제 고지:** 본 문서는 전략 설계이며 법률 자문이 아니다. 실제 분쟁 가능성이 구체화되면
> 한국 변호사 검토를 별도로 받는다. 본 문서는 "선의(good faith)로 정당하게 설계했다"는 **증거 자체**로도 기능한다.

## 1. 확정된 데이터 저장 정책 (선생님 승인 2026-06-07)

> 원칙: **"사실(fact)은 저장한다. 식별 민감정보는 보유하지 않는다."**
> 미성년/성인 무관 — **팩트 기반 기록은 동일하게 저장.** (선생님 명시)

| 데이터 | 분류 | 저장 | 화면노출 | 비고 |
|---|---|---|---|---|
| 경기 기록(누가·언제·종목·기록·순위) | 사실 | ✅ | ✅ | 서비스 본체. 가장 안전 |
| 이름 | 공개경기 출전정보 | ✅ | ✅ | 공개 범위 내 |
| 소속(연도별) | 공개 사실 | ✅ | ✅ | 소속이력 = 사실의 일부 |
| 대회·날짜·풍속·신기록여부 | 사실 | ✅ | ✅ | |
| 생년(2자리, '95) | 부분 식별정보 | △ 최소 | △ 동명이인 구분시만 | 식별력 약함 |
| **생년월일(full)** | 민감 개인정보 | ❌ | ❌ | 불필요·고위험 |
| **person_no(고유번호)** | 타기관 부여 식별자 | ❌ **(B안)** | ❌ | §2 참조 |
| 연락처·주소·주민번호 | 민감 개인정보 | ❌ | ❌ | 절대 미수집 |

> **미성년 정책 (제 권고 → 선생님 위임):** 기록(사실)은 성인과 동일 저장·노출.
> 단 **민감 부가정보(생년월일·연락처 등)는 성인보다 더 강하게 배제.** 삭제요청은 보호자도 가능하게.
> (해외 Athletic.net 등은 고교생을 노출하나, 한국 법·정서가 더 엄격하므로 "사실만" 원칙을 미성년에 특히 보수적으로 적용.)

## 2. person_no "판단 후 폐기" (B안) — 확정

```
[수집 시점]  외부 출처에서 동일인 후보 비교 (예: 2022 ○○고 홍길동 ↔ 2024 △△대 홍길동)
              └─ person_no(또는 등가 식별자)는 "동일인 판단" 연산에만 메모리상 일시 사용
[판단]       동일인으로 확정되면 → 우리 자체 난수 canonicalId 발급 (예: at_a8f3e1...)
[저장]       DB에는 (canonicalId ↔ 기록들) 관계만 저장.
              person_no 원본·생년월일은 **저장하지 않음 / 즉시 폐기**
[결과]       소속 바뀐 기록이 한 선수로 묶임 ✅  +  우리 DB에 타기관 개인정보 없음 ✅
```

방어 효과:
- "person_no 무단 DB 구축" 혐의 → 우리 DB에 person_no가 **존재하지 않음**으로 소명.
- canonicalId는 **역추적 불가 난수**(person_no와 수학적 연관 없음). 우리 웹을 누가 긁어도 person_no 복원 불가.
- 처리 코드·폐기 로그를 남겨 "정말 안 남긴다"를 입증 가능하게 한다.

> **보안 강화(선생님 요청 "우리끼리, 긁어도 안 보이게"):**
> ① canonicalId는 솔트 해시 기반 난수. ② API 응답·HTML·JSON 어디에도 person_no/생년월일 없음.
> ③ 매핑 연산은 서버 내부 전용 모듈에서만. ④ 매핑 작업 산출 중간파일은 작업 종료시 삭제.

### 2.1 🚫 person_no 기반 "전체 선수 정제" 금지 — 확정 (선생님 승인 2026-06-08)

B안의 핵심은 "person_no를 저장하지 않는다"이지만, **"person_no로 전체 데이터를 일괄 정제한다"는 것도 별개로 금지**한다.
실측(2026-06-08, 91,004행) 결과 문제의 본질은 **진학/이적(712명)이 아니라 동명이인(강한 신호 4,735명, 의심 행 77,286행, 약 86.2%)**이다.
즉 "고유번호만 있으면 다 정리된다"가 아니라, **이름 기반 자동 병합은 거의 하면 안 된다**가 맞다. 미성년 비율이 높아 잘못된 병합은
단순 오류가 아니라 **"타인의 기록을 다른 사람에게 붙이는 문제"**가 된다.

따라서 다음을 마스터 원칙으로 못박는다:

| 항목 | 정책 |
|---|---|
| 동명이인 | **합치지 않는다.** 이름이 같아도 한 사람으로 자동 병합 ❌ |
| 진학/이적 추정(712명) | 확정 병합 ❌ → **`추정 묶음` 또는 `opt-in 안전병합`**으로만. 기본 표시는 "분리" |
| person_no Layer3 | **전체 적용 ❌.** 검색 빈도 높은 선수·본인 요청·매거진 콘텐츠용 **고가치 소수만** |
| 전체 일괄 정제 | **금지.** 끝없는 정제에 리소스를 빨리지 않는다 |
| "내 기록 묶기" 니즈 | 자동 병합이 아니라 **본인 인증/요청 기반**으로 해결 |
| 파싱쓰레기 이름(`"0"` 등) | **검색/통계/인사이트 색인에서만 제외**, `data/results/*.json` 원본은 보존 |

> 상세 ROI 근거·실측 수치: `athletetime-identity-roi-decision.md`
> 본인 요청 기반 묶기 흐름: `athletetime-self-claim-record-grouping.md`
> 정체성 3레이어 구조: `athletetime-athlete-identity-architecture.md`

> **이 결정의 의미:** AthleteTime은 "공식 선수 DB인 척하는 서비스"가 아니라
> **"신뢰 가능한 육상 공개기록 검색·해석 서비스"**로 남는다. 향후 고유번호를 붙이더라도 구조를 무너뜨리지 않고 Layer3로 제한 확장.

## 2.2 기존 보유 데이터(2018~2026)의 정당성 — 소급 입장 확정 (2026-06-12)

> 배경: 현재 서비스가 보유·노출 중인 64,325건은 과거 세션에 `result.kaaf.or.kr`(robots `Disallow:/`)에서
> 수집된 것이다. "신규 수집 동결"(legacy-collector-policy)과 별개로, **이미 보유한 데이터를 계속 노출·정리하는 것**에 대한
> 입장을 분명히 둔다. (선생님 판단 2026-06-12: "결국 공공 사실을 모은 것이라는 원론이면 큰 문제 없다")

| 쟁점 | 확정 입장 |
|---|---|
| **무엇을 보유하나** | 원본 페이지의 복제·미러링이 아니라, **개별 경기 사실(누가·언제·어디서·몇 등·기록)만 추출해 독자 스키마로 재구성**한 것. 실측: 저장 데이터에 원본 HTML 구조·person_no·생년월일 **0건**, `{rank, name, affiliation, record}`만 존재 → "사실 재구성물"임이 데이터로 증명됨. |
| **robots 모순 해소** | robots는 **자동 크롤러의 신규 접근**을 규율하는 규약이다. 이미 취득한 공개 사실의 **보유·정리·노출을 소급 금지**하는 것이 아니다. 우리는 신규 크롤링을 동결(코드 가드)했고, 보유분은 사실 재구성물로 운영한다. → "동결했는데 왜 쓰나"는 자기모순이 아니다. |
| **출처 표기 (#4)** | "제공: 대한육상연맹"처럼 **연맹이 능동적으로 제공했다는 오인**을 주지 않는다. **"출처: …공개 경기결과 · {대회}({날짜}) — AthleteTime이 모아 정리"**로, *우리가 공개 자료에서 모았다*는 사실 그대로 표기. (구현: `frontend/src/config/dataPolicy.ts` `formatSource`) |
| **안전장치** | 과하게 방어적으로 가지 않는다(선생님 판단). 분쟁 시 즉시 비노출은 기존 `dataRequestService` suppression으로 충분. 출처별 일괄 비노출은 운영 절차로만 둔다(무거운 코드 스위치 불필요). |

> **결론:** 본 서비스는 "공개된 경기 사실을 모아 정리한 자료"다. 이는 저작권법상 보호되지 않는 *사실 그 자체*이며,
> 해외 동종 산업(§4)에서 합법적으로 운영되는 표준 행위다. 이 입장을 이용자에게도 `/about-data` 페이지로 공개 선언한다(#9).

## 3. 다중 방어선 (Defense-in-Depth) — 6겹

분쟁 시 "위법 단정"을 막고 협상/소명 단계로 끌고 가는 구조. 각 방어선은 독립적이다.

| # | 방어선 | 무엇을 막나 | 구현 방식 |
|---|---|---|---|
| **①** | **사실/편집물 분리** | 저작권·DB제작자권(§93) | KAAF 구조 미러링 금지. 개별 사실만 추출→우리 스키마로 재구성 |
| **②** | **person_no 비보유(B안)** | 개인정보보호법 | 판단 후 폐기, 난수 canonicalId만 저장 |
| **③** | **검색엔진/색인자 프레임** | 약관 영리·재배포 조항 | robots 준수 + 출처링크 의무 + "공식 아님" 고지 |
| **④** | **즉시 삭제 SLA** | 분쟁 동기 자체 | suppression 재사용, 요청 후 N시간 내 색인 제외 명문화 |
| **⑤** | **공공데이터 우선 골격** | 출처 정당성 | data.go.kr(상업이용 허가) 데이터를 1차 골격으로 |
| **⑥** | **업계 표준관행 근거** | "이상한 짓" 프레이밍 | 해외 동종서비스 다수 존재(§4)를 정당성 논거로 |

> **순서 원칙(중요):** 방어선은 **"빠져나가려는 회색지대"가 아니라 "진짜 권리존중에서 자연히 생기는 회색지대"** 여야 통한다.
> 일부러 모호하게 만든 정황(악의)이 드러나면 역효과. → 우리는 **실제로 권리를 존중하는 설계**를 하고, 그 결과 방어력이 생긴다. 이 순서를 문서로 못박는다.

## 4. 해외 사례 — 정당성의 근거 (실측 2026-06-07)

"기록 검색·비교·선수추적"은 **글로벌 표준 산업**이다. 우리가 하려는 것은 예외적 행위가 아니다.

| 서비스 | 규모/성격 | 시사점 |
|---|---|---|
| **Tilastopaja** (핀란드) | **선수 1,289,341명 / 기록 17,560,737건**, 유료구독(영리) | 영리 기록DB가 합법 운영. 우리 모델의 선례 |
| **World Athletics** 공식 | 선수프로필+DOB+랭킹 공개, **공식 Timing API 제공** | 공식기관도 데이터 공개·API화 |
| **OpenTrack** | WA API로 PB·랭킹 조회 연동 | 공식 API 연동형 정공법 사례 |
| **TFRRS** (미국) | 대학 육상 랭킹·기록 검색 DB | 교육기관 단위 공식 연동 |
| **Athletic.net / MileSplit** | 고교 포함 결과·시즌비교 | 학생선수 포함(단 미국 법제) |
| **DirectAthletics** | "연 수백만 결과 수집, **자동 선수추적(athlete tracking)**" | 우리 canonicalId 묶기 = 업계 표준 기능 |
| **alltime-athletics.com** | 개인 운영 역대기록 리스트 | 개인도 사실DB 운영 가능 |
| **Kaggle WA 50만건 데이터셋** | 사실 데이터 2차 공개·재활용 | 사실의 재배포 사례 |

> **단, 관할 차이 주의:** 해외(미·유럽)는 개인정보·미성년에 상대적으로 관대하고 연맹이 공식 API를 주는 경우가 많다.
> 한국은 ① 개인정보보호법이 더 엄격, ② 미성년 보호 정서·법 강함, ③ KAAF 공식 API 부재.
> → **"해외에 있으니 개념·기능은 정당하다"는 ✅ 쓰되, "그러니 한국 개인정보·미성년 규정도 무시"는 ❌.**

## 5. 데이터 출처 전략 (우선순위 + 관할)

```
1순위 (골격, 합법성 확실)
   └ 한국 공공데이터포털 data.go.kr  — 선수등록/팀/경기참가 (상업이용 허가, 익명통계)
   └ World Athletics 공식 통계/API   — 국제 선수, 공식 제공분
2순위 (색인, 정도 6원칙 준수)
   └ www.kaaf.or.kr 공개 페이지      — robots 허용 구역만, 사실만 추출
   └ 해외 공개 통계 사이트의 "사실"   — 사실 인용 범위, 출처링크
제외 (건드리지 않음)
   └ result.kaaf.or.kr               — robots 전면차단(Disallow:/)
   └ pis2.sports.or.kr               — 관리자층, 서비스 범위 밖
   └ 실시간/유료제한 구역             — 제한 명시된 곳
```

### 5.1 해외 데이터 수집 경로 (선생님 "해외 수집 방법 고려" 반영)
- **(권장) 공식 API 우선:** World Athletics Timing API 등 공식 제공 경로가 있으면 그걸 1순위로.
  공식 API 사용은 약관 동의 하에 합법적이며 가장 안정적. (OpenTrack 선례)
- **공개 데이터셋:** Kaggle 등에 공개·라이선스 명시된 사실 데이터셋 활용 — 출처·라이선스 보존.
- **관할(jurisdiction) 고려:** 해외 사이트는 해당국 법·robots·약관을 따른다.
  EU 출처는 GDPR, 미국 출처는 각 주법. → 사실(fact)만 취하고 개인식별정보는 동일하게 배제.
- **주의:** "해외 서버에서 수집하면 한국법 회피" 식 접근은 ❌. 우리 서비스가 한국 이용자 대상이면
  한국 개인정보보호법이 적용될 수 있다. 관할 우회를 방어전략으로 삼지 않는다(악의 정황 회피).
