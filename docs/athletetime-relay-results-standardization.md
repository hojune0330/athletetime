# 역전(구간)마라톤 결과 규격화 스펙 — Codex 작업지시서

> 담당: **Codex** / 발행: 2026-07-06 / 상태: **OPEN — 수정 전까지 해당 대회는 사용자 노출 금지 대상**
> 검증 도구: `node scripts/validate-relay-results.js` (이 문서와 한 몸. 통과 = 완료 조건)

## 1. 증상 (사용자 신고)

`/competitions` 대회 결과에서 **코오롱구간마라톤대회**를 열면 선수명·내용이 전부 틀려 보인다.

실제 데이터 (`data/results/2026.json`, `2026-road-005` 남자중학교부):

```json
{ "rank": 1,
  "name": "박준혁(7회)건대부중  10:39",
  "affiliation": "오성일(10회)배문중  12:02",
  "record": "" }
{ "rank": 2,
  "name": "곽민재11:13  238:44",
  "affiliation": "오대찬0  0",
  "record": "" }
```

- `name`에 **다른 주자의 이름+구간기록**이 붙어 있음
- `affiliation`에 소속이 아니라 **또 다른 주자**가 들어 있음
- `record`가 비어 있거나(중등부) **엉뚱한 주자 문자열**(고등부)임
- `48:44`가 `238:44`처럼 붙는 등 셀 병합 텍스트가 그대로 이어짐

## 2. 원인 진단 (분석 완료 — 재조사 불필요)

역전(구간) 대회의 KAAF 결과 테이블은 일반 트랙 종목과 구조가 다르다:

- **일반 트랙**: 1행 = 선수 1명 → `순위 | ... | 이름 | 소속 | 기록 | ...`
- **역전경기**: 1행 = **팀 1개**, 열이 **구간(1구간, 2구간, …)별 주자 셀**로 반복되고,
  각 셀 안에 `이름(회차)소속 구간기록` + 누적기록이 뭉쳐 있음. rowspan/colspan 병합 존재.

그런데 `src/scraper.js` (≒ `card-studio/scraper.js`) 트랙 분기(≈ L345-375)는 **고정 인덱스**
`cells[3]=name, cells[4]=affiliation, cells[5]=record`로 읽는다. 역전 테이블에 이 로직이
그대로 적용되어 구간 주자 셀들이 이름/소속/기록 필드에 잘못 매핑된 것.

## 3. 영향 범위 (전수 스캔 결과 — `\d{1,2}:\d{2}` 패턴이 name에 섞인 행 기준)

- 공개 결과 게이트(`validate-relay-results.js`)는 **공개 대상 25건**만 검사한다.

## 4. 목표 규격 (역전 결과 스키마)

역전경기는 "팀 순위 + 구간별 주자"가 본질이다. 기존 개인 스키마에 우겨넣지 말고 팀 단위로 규격화한다:

```jsonc
// events[].results[] — 역전경기 전용 (tableType: 'relay')
{
  "rank": 1,
  "team": "배문중",              // 팀명 (필수)
  "record": "2:38:44",           // 팀 최종(누적) 기록 (필수, HH:MM:SS 또는 MM:SS)
  "note": "",
  "newRecord": "",
  "legs": [                       // 구간별 주자 (선택이지만 있으면 전부 규격 준수)
    { "leg": 1, "name": "박준혁", "affiliation": "건대부중", "legRecord": "10:39" },
    { "leg": 2, "name": "곽민재", "affiliation": "배문중",   "legRecord": "11:13" }
  ]
}
```

규격 규칙 (검증 스크립트가 강제):
- R1. `name`/`team`/`affiliation`에 시간 패턴(`\d{1,2}:\d{2}`) 또는 3자리 이상 연속 숫자 금지
- R2. `record`는 시간 형식(`H:MM:SS` | `HH:MM:SS` | `MM:SS`) 또는 상태 표기(DNF/DNS/DQ)만 허용
- R3. `legs[].name`은 한글/영문 이름만 (괄호 회차 표기는 제거), `legRecord`는 시간 형식
- R4. 팀 rank는 1부터 연속 (동순위 허용), 중복 행 금지
- R5. 원본에서 파싱 확신이 없는 행은 **버리지 말고** `"parseStatus": "unverified"`로 표시
      → 프론트는 unverified 행을 노출하지 않음

## 5. 작업 항목 (순서대로)

### A. 스크레이퍼에 역전 테이블 분기 추가 — `src/scraper.js` + `card-studio/scraper.js` (동일 반영)
1. 테이블 유형 감지: 종목명에 `역전|구간` 포함 or 헤더에 `구간` 열 존재 → `tableType: 'relay'`
2. 역전 파서: 팀 행 단위로 `rank/team/record` 추출, 구간 셀은 `이름(회차)소속  시간` 패턴 분해
   - 셀 텍스트 정규식 예: `/^(.+?)(?:\((\d+)회\))?([가-힣A-Za-z\s]+?)\s+(\d{1,2}:\d{2}(?::\d{2})?)$/`
   - rowspan/colspan 병합은 DOM 기준으로 처리 (textContent 이어붙이기 금지 — 지금 사고의 원인)
3. 파싱 실패 행은 R5의 unverified 처리 (절대 기존처럼 오염된 채 저장 금지)

### B. 기존 오염 데이터 재수집 또는 원본 대조 정정
- 공개 대상 25건을 원본(각 대회 KAAF 결과 페이지/결과 파일)에서 재수집 → 새 파서로 재생성
- 원본 접근 불가(연도 오래됨 등) 항목은 해당 event의 results를 비우고
  `"resultsStatus": "source_reverify_needed"` 마킹 → 프론트에서 "기록 정리 중" 표시
- 절대 금지: 오염 데이터를 그대로 두고 프론트에서만 가리는 미봉책

### C. 노출 차단 가드 (파서 수정과 독립적으로 즉시)
- `card-studio/services/dataQualityService.js`에 R1 위반 감지 추가
- 위반 결과는 `/api/competitions/:id` 응답에서 해당 event `results`를 숨기고
  `"qualityHold": true` 반환 → 프론트는 "기록 확인 중이에요" 안내
- 검색 인덱스(`searchService`)·프로필카드 프리필에서도 R1 위반 이름 제외
  (이미 `isIndexableAthleteName`이 일부 거를 수 있으니 확인 후 보강)

### D. 검증 게이트 등록
- `node scripts/validate-relay-results.js` → 전 연도 R1~R4 위반 0건이면 exit 0
- 통과 후 `backend/tests/relay-results-standard.test.js` 계약 테스트 작성해서
  package.json test 스크립트에 등록 (스크립트 호출 + 스키마 고정)
- 최종: `npm test` 전부 통과 + `npm --prefix frontend run build` 그린
