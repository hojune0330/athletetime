# AthleteTime Auth, Privacy, Security Contract

이 문서는 AthleteTime 회원가입, 로그인, 커뮤니티, 계정 정보 처리의 출시 차단 기준이다. 기능을 빠르게 늘리더라도 사용자 계정과 개인 활동을 지키는 기준은 이 문서보다 약해질 수 없다.

## 1. 목표

- 선수와 러너가 자기 이름, 기록, 커뮤니티 활동을 안심하고 다룰 수 있게 한다.
- 운영자가 계정, 신고, 정정 요청을 처리할 때 같은 기준으로 판단하게 한다.
- “법적으로 안전하다”는 홍보 문구가 아니라, 수집 최소화와 삭제 가능한 구조로 신뢰를 만든다.

## 2. 세션과 토큰

- 실사용 전환 시 로그인 세션은 `HttpOnly`, `Secure`, `SameSite` 쿠키를 기본값으로 한다.
- 인증 토큰, 재발급 토큰, 관리자 토큰은 `localStorage` 또는 `sessionStorage`에 저장하지 않는다.
- 현재 프론트엔드의 `localStorage` 토큰 방식은 임시 구조다. 운영 전환 작업에서 반드시 제거한다.
- 로그아웃은 서버 세션 무효화와 브라우저 쿠키 제거를 함께 수행한다.

## 3. CSRF와 요청 보호

- 쿠키 기반 인증으로 전환할 때 모든 쓰기 요청은 `CSRF` 토큰 또는 동등한 방어책을 사용한다.
- 커뮤니티 글쓰기, 댓글, 좋아요, 프로필 변경, 계정 삭제, 데이터 요청 상태 변경은 보호 대상이다.
- CORS는 운영 도메인 allowlist로 제한한다. 개발 편의를 위한 전체 허용 설정은 운영에서 금지한다.

## 4. 이메일 인증과 비밀번호 재설정

- 회원가입 인증 코드와 비밀번호 재설정 인증 코드는 서버 로그, 브라우저 콘솔, 오류 응답에 남기지 않는다.
- 비밀번호 찾기는 이메일이 가입되어 있는지 외부에 드러내지 않는다. 등록된 이메일과 미등록 이메일 모두 같은 공개 응답을 반환한다.
- 관리자 승격 기능은 `ADMIN_SECRET_KEY`가 명시적으로 설정된 환경에서만 동작한다. 과거 기본키나 빈 값은 허용하지 않는다.
- 인증 코드, 임시 비밀번호, 재설정 링크, 세션 토큰은 운영자가 평문으로 확인할 수 없어야 한다.

## 5. 계정과 개인정보 권리

- 사용자는 회원 탈퇴를 요청할 수 있어야 한다.
- 사용자는 자기 계정 정보와 커뮤니티 활동 내역의 내보내기를 요청할 수 있어야 한다.
- 사용자는 커뮤니티 닉네임, 프로필, 공개 활동에 대한 삭제 또는 익명화를 요청할 수 있어야 한다.
- 공개 경기 기록의 정정, 비노출, 삭제 요청은 계정 삭제와 분리해서 처리한다. 경기 기록은 데이터 출처와 공익성, 미성년 보호, 본인 요청 여부를 함께 본다.

## 6. 보관 기간과 삭제 기준

- 인증 코드는 짧은 만료 시간을 두고 만료 뒤 재사용할 수 없게 한다.
- 비밀번호 재설정 코드, 이메일 인증 코드, 세션 기록은 필요한 보관 기간이 지나면 삭제한다.
- 보안 사고 조사에 필요한 최소 로그만 남긴다. 로그에는 비밀번호, 인증 코드, 토큰, 원본 식별자, 민감한 개인정보를 남기지 않는다.
- 탈퇴한 계정의 커뮤니티 글은 서비스 안정성과 맥락 유지를 위해 익명화할 수 있다. 다만 법적 요청, 미성년 보호, 명예훼손 우려가 있는 경우 삭제를 우선 검토한다.

## 7. 공개 기록과 계정 데이터의 분리

- 공개 경기 기록 색인은 회원 계정과 자동으로 연결하지 않는다.
- 같은 이름의 기록은 기본적으로 분리해서 보여준다. 사용자가 본인 기록 묶기를 요청한 경우에만 별도 확인 흐름을 둔다.
- 계정 이메일, 로그인 이력, 커뮤니티 활동을 경기 기록 정제나 선수 식별에 사용하지 않는다.
- 선수 고유번호, 생년월일, 원본 식별자는 전체 정제 목적으로 저장하지 않는다는 기존 데이터 전략을 따른다.

## 8. 관리자와 운영자 접근

- 관리자 화면은 인증된 관리자만 접근할 수 있어야 한다.
- 관리자 승격, 요청 처리, 삭제, 복구, 차단 같은 행위는 감사 로그를 남긴다.
- 감사 로그는 누가 언제 무엇을 했는지 확인하는 용도이며, 인증 코드나 비밀번호 같은 비밀값을 포함하지 않는다.
- 운영자는 이용자에게 내부 판단 로직, 보안 대응 세부 절차, 취약점 힌트를 공개하지 않는다.

## 9. 금지 원칙

- 사용자에게 법적 안전을 단정하거나 보장하지 않는다.
- 자동화나 모델이 공식 판단을 대신한다고 말하지 않는다.
- 계정 데이터와 공개 기록을 몰래 연결하지 않는다.
- 가입 여부, 관리자 여부, 삭제 요청자 신원처럼 악용 가능한 정보를 오류 메시지로 노출하지 않는다.

## 10. 출시 차단 기준

다음 조건 중 하나라도 남아 있으면 실사용 커뮤니티 출시를 보류한다.

- 인증 토큰이 `localStorage`에 저장된다.
- 쿠키 세션 전환 뒤 쓰기 요청에 `CSRF` 방어가 없다.
- 인증 코드, 재설정 코드, 토큰, 비밀번호가 로그에 남는다.
- 비밀번호 찾기 응답으로 가입 여부를 알 수 있다.
- 회원 탈퇴, 내보내기, 삭제, 익명화 요청 경로가 없다.
- 관리자 승격이 기본키, 빈 값, 예측 가능한 값으로 가능하다.
