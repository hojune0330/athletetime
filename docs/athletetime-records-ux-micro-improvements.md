## A등급 — 단독 착수 가능 (프론트 단독)

### A1. 검색 입력 접근성(a11y) 보강 — **최우선 추천**
**현황:** `RecordsPage`에 `aria-label`/`<label>`/`aria-live`/`sr-only`가 **하나도 없음**(grep 0건).
- 검색 인풋(207행)에 라벨 없음 → 스크린리더가 "선수 이름, 소속, 종목"을 placeholder로만 노출(접근성상 불충분).
- 검색 결과/오류/빈 상태 전환이 `aria-live`로 안내되지 않음 → 비시각 사용자는 결과 갱신을 인지 못 함.
- "2글자 미만이면 버튼 disabled" 규칙(214행)이 시각적으로만 존재 → 왜 못 누르는지 안내 없음.

**제안(코드 무변경, 후보):**
- `<Input aria-label="공개 기록 검색">` + 결과 컨테이너에 `aria-live="polite"`.
- 빈/오류 `NoticeCard` 영역을 `role="status"`로.
- 버튼 옆 또는 도움말로 "두 글자 이상 입력하면 검색할 수 있어요" 보조 문구(`text-xs text-ink-4`).

**리스크:** 매우 낮음(시맨틱 속성 추가). 신뢰 톤 영향 없음.
**예상 변경:** RecordsPage 검색 폼 영역 5~8줄.

---

### A2. 검색 디바운스 + 입력 즉시 피드백
**현황:** `handleSearch`는 submit(폼 제출/버튼) 기반. 타이핑 중 후보가 미리 안 보임 → 한 번 더 눌러야 하는 마찰.
**제안:** 입력 2글자 이상 시 300~400ms 디바운스로 후보 미리보기(서버 추가 호출 없이 기존 검색 API 재사용). submit은 유지(엔터/버튼).
**리스크:** 중. API 호출 빈도↑ → Codex와 호출량 합의 권장(B로 승격 가능). 우선은 "후보"로만.
**판단:** A2는 호출량 영향 때문에 **B로 재분류**가 안전. 아래 B3 참조.

---

### A3. 시즌 기록표 — 첫 열(선수) sticky + 모바일 가독
**현황:** 588행 `min-w-[720px]` 가로 스크롤 표. 모바일에서 가로로 밀면 "선수" 열이 사라져 어떤 행인지 추적 어려움.
**제안:** `<td>` 선수 셀 `sticky left-0 bg-...`로 고정, `thead`도 `sticky top-0`(스크롤 컨테이너 내). 기록 값 `font-mono`/정렬 유지.
**리스크:** 낮음(표 스타일만). 신뢰 톤 무관.
**예상 변경:** 표 헤더/첫 열 className 4~6줄.

---

### A4. 빈/오류 상태 카피 일관화 + 행동 유도
**현황:** "찾는 기록이 아직 없어요 / 이름이나 소속을 바꿔보세요"(246행) 등은 톤 양호하나 **다음 행동 버튼이 없음**.
**제안:** 빈 상태에 "시즌 기록표로 둘러보기"(이미 있는 `season` 모드 전환) 보조 액션 1개 추가. 미래 약속어("곧/예정") 금지 준수.
**리스크:** 낮음. 라우트 무변경(모드 토글 재사용).
**예상 변경:** NoticeCard children에 버튼 1개.

---

### A5. "여기서 시작하세요" 3-step 카드 — 모바일 세로 정렬 확인
**현황:** 342~362행 `step={1/2/3}` 원형 배지(직전 라벨 리팩터 결과). 데스크탑은 양호.
**제안:** 모바일(<sm)에서 3카드가 세로로 쌓일 때 간격/번호 배지 정렬 점검만(스크린샷 회귀). 코드 변경은 필요 시에만.
**리스크:** 매우 낮음(검증 항목).

## B등급 — 합의 필요 (공용/라우트/호출량 영향)

### B1. 검색 → 결과 진입 시 URL 쿼리 동기화(`?q=`) 일관성
**현황:** 홈에서 `/records?q=...`로 진입하는 경로 존재. RecordsPage 내부 검색이 URL과 100% 동기화되는지(뒤로가기/공유) 점검 필요.
**합의 포인트:** 라우팅/히스토리 동작은 공유 UX → 사용자 확인 권장.

### B2. 비교 트레이(CompareView) 모바일 하단 고정 — 탭바와 충돌 검토
**현황:** 326행 `compareTray` 하단 여백(`h-28 sm:h-24`) 확보 중.
**합의 포인트:** Option 1 **하단 탭바** 도입 시 비교 트레이와 z-index/위치 충돌. **탭바 합의와 묶어서** 결정해야 함.

### B3. 검색 디바운스/타입어헤드 (A2에서 승격)
**합의 포인트:** 검색 API 호출량↑ → Codex(백엔드/데이터)와 호출 빈도·캐싱 합의 필요.
