## 1. 존재 이유 — 한 문장

**"내 육상 기록이 어딘가에 남아 있다는 걸, 내 이름 하나로 확인할 수 있는 유일한 곳."**

한국 육상은 공개 경기결과가 대회별 PDF·게시판·개별 사이트에 흩어져 있다.
선수(특히 학생·동호인)는 자기 기록의 "흐름"을 볼 방법이 사실상 없다.
AthleteTime은 그 흩어진 공개 기록을 모아 **이름 → 후보 확인 → 기록 흐름**으로 30초 안에 보여준다.
이것이 다른 무엇으로도 대체되지 않는 핵심 가치다. 나머지 기능은 전부 이 루프의 보조다.

### 무엇이 아닌가 (지키는 선)
- 공식 기록·랭킹 기관이 아니다. 순위·평가·예측을 만들지 않는다.
- 커뮤니티 포털·만능 육상 앱이 아니다. (커뮤니티는 기록 이야기의 파생)
- 개인정보 저장소가 아니다. 생년월일·연락처·식별번호를 모으지 않는다.

## 2. 사용자가 다시 오는 이유 — 3개의 루프

### 루프 A. 대회 사이클 루프 (계절성 재방문 — 가장 강함)
```
대회 일정 확인 → 대회 당일 결과 확인 → 내 기록 갱신 확인 → 다음 대회 일정 확인
```
- 육상 선수의 삶은 대회 캘린더를 따라 돈다. 서비스도 이 리듬을 그대로 탄다.
- 홈 상단 "지금 열리는 대회 / 다가오는 대회" 카드가 이 루프의 입구. (이미 구현됨)
- **첫 주 목표**: 대회가 끝난 뒤 "결과가 올라왔어요"가 아닌 실제로 기록이 검색되는 경험 1회.
  이 1회가 생기면 "다음 대회 후에도 여기서 확인하면 되겠다"는 습관 가설이 성립한다.

### 루프 B. 기록 정체성 루프 (개인적 재방문)
```
내 이름 검색 → 기록 흐름(PB·시즌·최근) 확인 → 기록카드로 공유 → 지인이 링크로 유입 → 그 지인도 자기 이름 검색
```
- 공유되는 것은 자랑이 아니라 "확인 가능한 사실"이다 — 출처·워터마크가 함께 간다.
- 기록카드·공유 링크가 유일한 유기적(organic) 성장 채널. 광고 없이 퍼질 수 있는 구조.
- **첫 주 목표**: 검색 → 선수 상세 → 공유(링크 또는 카드)의 전환 마찰 최소화. (이번 PR의 네비 단순화가 이 목적)

### 루프 C. 이야기 루프 (감정적 재방문 — 장기)
```
기록에서 이야깃거리 발견 → 커뮤니티에 짧은 글 → 반응 확인하러 재방문
```
- 커뮤니티는 독립 포털이 아니라 기록의 파생물로 유지한다(기록 컨텍스트 프롬프트 이미 구현).
- 첫 주에는 강요하지 않는다. 글 0개인 커뮤니티를 메인에 크게 걸면 신뢰가 깎인다.

## 3. 첫 주 신뢰 원칙 — "정직함이 곧 UX"

첫 방문자가 10초 안에 판단하는 것: ① 이게 뭐 하는 곳인지 ② 믿을 수 있는지 ③ 내 것이 있는지.

| 원칙 | 구현 위치 |
|---|---|
| 무엇을 모았고 무엇이 아닌지 검색 **전에** 고지 | 홈 히어로 신뢰 앵커 (이번 PR) |
| 완성 안 된 기능을 완성된 것처럼 보이지 않게 | 헤더 1차 네비 축소 + 더보기 그룹 (이번 PR) |
| 길을 잃어도 죽은 끝이 없게 | 404 → 기록 검색 복구 (이번 PR) |
| 빠진 기록·동명이인 가능성을 숨기지 않기 | TRUST_NOTICE (기존 구현 유지) |
| 틀린 정보를 사용자가 직접 고칠 수 있게 | /data-request 노출 강화 (이번 PR 홈 앵커) |

**금지**: 공식/랭킹/검증/예측/평가 표현, 미래약속("곧", "예정"), 전국 완전 DB 암시.

## 4. 이번 PR에서 실제로 개선한 것 (🟠 High · 오픈 첫 주)

### 4-1. 헤더 IA 재구성 — 기능 박물관 → 핵심 루프
- **문제**: 1차 네비 9개(커뮤니티/기록카드/기록/대회/라이브/계산기/훈련/마켓/채팅).
  첫 방문자는 "이 서비스가 뭘 하는 곳인지" 판단 불가 + 미완성 화면(채팅·마켓)으로 이탈.
- **개선**: 1차 네비를 핵심 루프 순서(기록 → 대회 → 기록카드 → 커뮤니티) 4개로 축소.
  나머지 5개는 "더보기" 드롭다운(데스크톱)·"도구·부가 기능" 그룹(모바일 드로어)으로 강등.
  각 항목에 성격 주석(PaceRise 연동/훈련 보조/사용자 거래/익명 대화)을 붙여 기대치를 관리.
- **효과**: 모바일 하단 탭바(홈/기록/대회/커뮤니티)와 데스크톱 네비의 정보구조 일치 → 학습 비용 1회.

### 4-2. 404 재설계 — 죽은 끝 제거
- **문제**: 이모지 + "URL을 다시 확인해주세요"뿐. 공유 링크가 데이터 정리로 깨졌을 때
  (실제로 발생하는 시나리오 — RecordsPage에 fallback 존재) 유입 사용자가 그대로 이탈.
- **개선**: 기록 검색 폼 내장 + 링크가 바뀌었을 가능성을 사람 언어로 설명 + 핵심 3경로 복구 링크.
  디자인 토큰을 다른 화면과 통일(구 neutral 팔레트 → ink/surface/brand).

### 4-3. 홈 히어로 신뢰 앵커
- **문제**: 검색을 유도하면서 "무엇을 보게 되는지"의 성격 고지가 스크롤 아래(운영 원칙)에만 있었다.
- **개선**: 검색창 바로 아래에 1줄 고지 + `어떻게 모았는지 보기(/about-data)` + `정정·비노출 요청(/data-request)`.
  검색 전 고지는 Notice & Takedown 방어 구조의 1층이기도 하다.

### 4-4. 푸터 정체성 문구 정합
- "대한민국 육상인을 위한 커뮤니티 플랫폼" → "흩어진 공개 육상 기록을 이름으로 찾는 곳".
  홈 히어로·SERVICE_IDENTITY와 포지셔닝 일치. (커뮤니티는 파생이지 정체성이 아님)

### 4-5. 회귀 잠금 — launch-surface 계약 테스트
- `backend/tests/launch-surface-nav.test.js` 신설 (5 tests):
  1차 네비 = 핵심 4개 고정, 보조 기능 1차 승격 금지, 더보기 접근성 보존,
  모바일 탭바 미러링, 404 복구 경로, 홈 신뢰 앵커.
- 이후 작업자가 "기능 하나만 네비에 추가"하는 무의식적 회귀를 테스트가 막는다.

## 5. 다음 작업자를 위한 우선순위 (오픈 첫 달)

### P1 — 첫 주 직후
1. **검색 실패 데이터 수집**: `/records` 검색어 중 0건 검색어를 익명 집계(개인정보 없음).
   빠진 대회·표기 변형(소속 접두 등)의 실사용 증거 → 데이터 파이프라인 우선순위 입력.
2. **대회 결과 신선도 라벨**: 대회 카드에 "며칠 전 수집" 표시 (TRUST_NOTICE.snapshot의 화면화).
   루프 A의 신뢰 핵심 — "결과가 언제 올라오는지"의 기대치 관리.
3. **프론트 빌드 청크 분할**: 이미 경고 중인 대형 청크. 모바일 첫 로딩이 루프 진입 장벽.

### P2 — 첫 달
4. **공유 링크 안정화**: athleteKey 변경 시 리다이렉트 맵 (지금은 404→검색 복구로 완화만 됨).
5. **시즌 기록표 SEO/공유 메타**: 검색엔진 유입이 유일한 무비용 획득 채널.
6. **커뮤니티 씨딩 정책**: 운영자 수동 씨딩 기준 수립 (자동 생성 금지 원칙 유지).

### 하지 말 것 (블루프린트 가드레일 재확인)
- 채팅/마켓/라이브를 1차 네비로 재승격 (launch-surface 테스트가 막음 — 풀려면 합의 먼저)
- 완전 DB·공식·랭킹 암시 카피
- 동명이인 자동 병합
- 예시/샘플 선수 데이터를 사용자 경로에 재도입

## 6. 성공 지표 제안 (첫 주 — 수동 확인 가능한 것만)

| 지표 | 왜 | 확인 방법 |
|---|---|---|
| 검색 → 선수 선택 전환 | 루프 B의 첫 관문 | 서버 로그 (검색 API → analytics API 순서) |
| 0건 검색어 수·유형 | 데이터 커버리지 실증 | P1-1 구현 후 집계 |
| 공유 링크 재유입 | 유기적 성장 신호 | `?athlete=` 파라미터 직접 진입 수 |
| /data-request 제출 | 신뢰 루프 작동 증거 (많아도 좋은 신호) | admin 페이지 |

정식 분석 도구 도입 전까지는 위 4개만 본다. 지표를 늘리는 것보다 루프 하나가 실제로 도는 것이 먼저다.
