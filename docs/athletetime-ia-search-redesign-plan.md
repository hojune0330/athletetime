> 목적: **실서비스 직전** 현재 웹의 과잉·중복·미반영 구조를 진단하고,
> "어차피 다시 수정하게 될 것"을 염두에 둔 재정비 방향과, 지금 골격에서 지켜야 할

## 0. 한 줄 요약
현재 검색·정보구조는 **3세대 구조가 겹쳐** 있다(구 .html 다중앱 → 구 SPA → 최근 refresh).
그래서 **검색 진입점이 3개로 분열**되어 있고 서로 다르게 동작하며,
**사이드바 메뉴 절반이 깨진 링크**다. 실서비스 전 "단일 검색 + 단일 IA"로 수렴해야 한다.

---

## 1. 진단 — 실측 증거 (2026-06-07)

### 1-1. 검색 진입점이 3중으로 분열 + 목적지 불일치 ⚠️ 최우선
| 진입점 | 위치 | 실제 동작 | 목적지 |
|---|---|---|---|
| Header 검색창 `SearchBar` | 모든 페이지 상단 | **`console.log`만** — 아무 동작 안 함 (`// 검색 로직 구현` 미완) | 없음 |
| 메인 검색폼 | `MainPage.tsx` `handleSearch` | navigate | `/competitions?tab=search&q=` (대회 게시판) |
| 기록 분석 검색 | `RecordsPage.tsx`(`/records`) | **진짜 검색** — `/api/.../analytics` 호출(athlete/season) | 자체 페이지 |

- **문제**: 사용자가 늘 보이는 상단 검색창에 입력해도 **아무 일도 안 일어남**.
  가장 강력한 `/records` 분석 검색에는 **어떤 검색창도 연결돼 있지 않음.**
- 같은 "선수·기록 검색"이 세 곳에서 다르게 동작 → 신뢰·일관성 붕괴.
- `SearchBar` "최근 검색어"(`100m 기록` 등)는 **하드코딩 더미**.

### 1-2. 사이드바 메뉴 = 구버전 IA (깨진 링크 다수) ⚠️
`components/layout/Sidebar.tsx` 가 App.tsx 라우트와 불일치:
| 사이드바 링크 | 실제 존재 라우트 | 상태 |
|---|---|---|
| `/events` (대회 게시판) | `/competitions` | ❌ 깨짐 |
| `/track` (종목별 게시판) | (없음) | ❌ 깨짐 |
| `/market` (중고거래) | `/marketplace` | ❌ 깨짐 |
| `/pace-calculator.html` 등 `.html` 직접링크 | `/pace-calculator`(SPA) | ⚠️ 이중구조 |
- "실시간 인기 키워드", "인스타그램 연동" = **더미/미연동**.

### 1-3. 페이지 과잉 (26개 top-level page)
커뮤니티/마켓/채팅/계산기/카드스튜디오/기록분석/관리자 등이 **한 평면에 평등하게** 나열.
- 핵심(기록 검색·아카이브)과 부가(커뮤니티·중고거래·계산기)의 **위계가 없음**.
- 서비스 정체성("기록 아카이브 + 검색엔진")이 메뉴에서 드러나지 않음.

### 1-4. 정체성·카피 혼재
"육상인들을 위한 통합 플랫폼"(만능 포털) vs 데이터 전략의 "사실 기반 기록 아카이브/검색엔진".
→ 신뢰 프레임(검색엔진·색인자)과 충돌하는 카피·더미가 곳곳에 남아 있음.

## 2. 재정비 목표 IA (To-Be) — 방향 합의용

### 2-1. 단일 검색 원칙 (Single Search Entry)
- **상단 검색창 = 유일한 1차 진입점.** 입력 → `/search?q=`(또는 `/records?q=`) 통합 결과로.
- 통합 결과 페이지에서 **탭/필터로 종류 구분**: `선수` · `기록(시즌표)` · `대회` · (후순위)`커뮤니티`.
- 더미(SearchBar console.log, 하드코딩 최근검색어) 제거. 모든 검색은 `/analytics`(사실) + 출처표기로.

### 2-2. 위계 있는 내비게이션 (3-tier)
```
1차(코어)  : 검색 · 기록(아카이브) · 선수
2차(도구)  : 페이스/훈련 계산기 · 카드 스튜디오
3차(커뮤니티/부가): 게시판 · 중고거래 · 채팅
```
- 사이드바 링크를 실제 라우트로 교정(`/events→/competitions`, `/market→/marketplace`, `/track` 제거/재정의).
- `.html` 직접링크 → SPA 라우트로 일원화.

### 2-3. 신뢰 프레임 일관화
- 모든 검색 결과 카드에 **출처·갱신시점·"공개기록 색인"** 표기(데이터 전략 6방어선과 일치).
- "통합 플랫폼" 류 만능 카피 → "공개 기록 검색·아카이브" 정체성으로 수렴.

## 3. 지금 골격에서 지킬 불변식 (재수정 비용 최소화 계약)
재정비가 올 것을 알기에, **지금 만드는 데이터/백엔드 골격은 UI에 종속되지 않도록** 한다.

1. **검색 백엔드 단일화**: 모든 검색은 `recordAnalyticsService` + `publicDataService`(거시통계)로만 응답.
   UI가 3개든 1개든 **백엔드 계약(JSON 형태)은 하나**. (현행 `/analytics/*` 유지)
2. **출처표기 필수 필드**: 검색 응답에 `source`(provider/agency/url/license) 동봉 — UI가 바뀌어도 보존.
3. **식별자 격리**: person_no 비보유(B안), canonicalId만. UI 재정비와 무관하게 유지.
4. **부가기능 결합도 0**: 커뮤니티/마켓/채팅은 코어 검색과 데이터 의존 없음 → 나중에 떼어내거나 숨겨도 코어 무영향.
5. **라우트 = 단일 출처**: 메뉴/링크는 App.tsx 라우트에서 파생(하드코딩 금지)하도록 @Codex와 합의.
